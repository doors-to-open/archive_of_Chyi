# Stage 6B Import Workflow

Stage 6B adds crawler and import tooling, but imported facts do not bypass review.

## Rule

Crawler output is a review draft. The crawler must not edit `data/*.json`.

Accepted records are added to the website data only after human review.

## Folders

- `scripts/import/`: tracked import and crawler code.
- `scripts/import/source-allowlist.json`: tracked public-source allowlist.
- `supplement/raw/`: ignored raw source snapshots.
- `supplement/import-drafts/`: ignored review candidate JSON files.
- `data/*.json`: reviewed archive records used by the website.

The `supplement/` folder is ignored by Git because it may contain temporary source snapshots, screenshots, and rejected drafts.

## Workflow

1. Choose one category and source family, such as releases from Apple Music or QQ Music.
2. Run the crawler against allowlisted public URLs.
3. Save raw responses under `supplement/raw/`.
4. Save candidate JSON under `supplement/import-drafts/`.
5. Review the candidate for source quality, duplicate records, missing fields, and uncertainty.
6. Add accepted facts to `data/*.json`.
7. Run checks/build.
8. Commit the accepted data batch and any import-tool updates.

## Review States

- `pending`: generated by crawler, not reviewed.
- `accepted`: reviewed and ready to add to `data/*.json`.
- `rejected`: not suitable for archive data.
- `needs-more-source`: useful lead, but not enough evidence.

## Candidate Shape

Each candidate should include:

- source URL and allowlist entry ID
- access date
- raw snapshot path when available
- HTTP status and content type when fetched
- parser notes
- extracted title, description, canonical URL, and platform metadata when available
- duplicate checks against existing data
- proposed source record
- proposed archive records, kept empty or partial until reviewed

## Source Priority

- Official pages, label pages, and official platform metadata are preferred.
- Encyclopedia and database pages are useful for orientation and cross-checking.
- Fan uploads can support media availability or setlist clues, but should usually remain `partial` or `uncertain` unless confirmed elsewhere.
- Anonymous or unsourced claims should not become confirmed archive facts.

## First Pass

Start with release/album candidates because album metadata has clearer structure and can be checked against the current `releases.json`, `songs.json`, and `sources.json` model.
